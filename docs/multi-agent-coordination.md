# 多智能体协作与动态规划 设计文档

## 1. 多智能体角色分工

系统包含四个职责严格隔离的 Agent，模拟真实面试中不同角色的信息边界：

| Agent | 职责 | 输入 | 输出 | 明确排除 |
|-------|------|------|------|----------|
| **Planner** | 统筹官：解析简历，生成面试议题 | 候选人信息 + 岗位描述 | 3-5 个结构化议题 | 不看到对话历史（面试尚未开始） |
| **Questioner** | 提问官：基于议题和历史生成追问 | 议题名称 + 全部对话历史 | 一道面试提问 | 不看到任何评分或评估结果 |
| **Evaluator** | 评估官：N=3 路并行独立评分 | 议题 + 最新一轮问答 | 评分 (1-100) + 理据 | 不看到完整对话历史，只看当前回答 |
| **Supervisor** | 仲裁官：人工介入断点 | 当前议题 + 候选人回答 + 分歧详情 | 人工决策 (CONTINUE/SKIP/END) | 不看到系统指令原文 |

**设计原则**：每个 Agent 只接收完成任务所需的最小上下文。Questioner 不知道评分，Evaluator 不知道历史对话——角色职责严格隔离，避免信息泄露和偏见传播。

## 2. 松耦合协作模式

Agent 之间**没有直接调用关系**，它们通过 `InterviewState` 共享状态进行通信：

```
┌─────────────────────────────────────────────────┐
│              InterviewState（共享）                │
│  candidate_info │ interview_plan │ chat_history  │
│  current_topic  │ evaluation_records │ routing   │
└──────┬──────┬───────────────┬──────────┬────────┘
       │      │               │          │
    Planner  Questioner    Evaluator  Supervisor
```

Agent 只读写共享状态，由 **Harness 层**（确定性代码）决定下一个激活哪个 Agent。这实现了"信息通过共享状态传递，流转由确定性逻辑控制"的松耦合协作模式。

## 3. 多路并行评估

Evaluator 是真正的"多智能体协作"场景——启动 N=3 个独立的评估器实例，对同一份回答独立打分：

```
候选人回答 → ┬─ Evaluator #1 → score=78, rationale="..."
             ├─ Evaluator #2 → score=82, rationale="..."
             └─ Evaluator #3 → score=45, rationale="..."
                  ↓
             σ = 16.2 → LOW → ESCALATE
```

三个评估器：
- **并行执行**，不互相等待
- 各自输出评分和理据，互不知晓对方结果
- 使用略有不同的 prompt 变体，避免同质化

这模拟了"多位面试官独立打分"的真实场景。

## 4. 基于治理指标的协作决策

多智能体协作的核心挑战是"意见不一致时怎么办"。系统用 σ（标准差）量化分歧：

| σ 区间 | 定性 | 协作动作 |
|--------|------|----------|
| σ ≤ 5.0 | 高度一致 | 记录平均分，继续下一议题 |
| 5.0 < σ ≤ 15.0 | 存在分歧 | 触发 CoT 重试或降维追问 |
| σ > 15.0 | 严重撕裂 | 丢弃成绩，交给 Supervisor 仲裁 |
| 连续 3 次中置信度 | 累计熔断 | 强制仲裁介入 |

这避免了"简单投票"或"固定权重平均"的粗糙做法，而是根据智能体的共识程度动态调整协作策略。

## 5. 面试过程的动态规划

面试不是线性执行的固定流程，而是根据候选人表现实时调整议题、策略和路径。

### 5.1 议题大纲的动态生成

Planner 不是用预设的固定题库，而是根据候选人简历和岗位描述**实时生成**议题：

```
候选人A（5年Python + 金融背景）
  → ["异步编程", "金融数据建模", "风控系统设计"]

候选人B（2年Java + 转Python）
  → ["Python基础", "语言迁移经验", "项目实践"]
```

议题数量（3-5个）也根据简历内容的丰富度动态决定。

### 5.2 路由决策的动态调整

系统根据实时评估结果动态选择最优路径：

```
候选人回答 → 3路评估 → σ 分析
  ├─ σ ≤ 5     (一致)  → 记录成绩 → 下一议题
  ├─ 5 < σ ≤ 15 (分歧) → 重试策略
  │   ├─ retry_count = 0  → 启用 CoT 重新评估
  │   └─ retry_count ≥ 1  → 降维追问（降低难度）
  └─ σ > 15    (严重撕裂) → 人工仲裁
```

### 5.3 难度的动态调整

中置信度重试时，系统会改变策略：

| 阶段 | 策略 | 说明 |
|------|------|------|
| 首次评估 | 标准 prompt | 正常评估 |
| 第1次重试 (CoT) | 启用思维链指令 | "请逐步分析候选人的回答" |
| 第2次重试 (降维) | 指示 Questioner 降低难度 | 把大问题拆解成更具体的小问题 |

这相当于面试中的"候选人没答上来，换个角度问"，难度曲线根据表现动态调整。

### 5.4 全局累积决策

`consecutive_medium` 计数器追踪连续中置信度次数：

| 情况 | 决策 |
|------|------|
| 第1次中置信度 | 重试，计数器 +1 |
| 第2次中置信度 | 重试，计数器 +1 |
| 第3次中置信度（累积熔断） | 强制 → 人工仲裁 |

单次分歧可以容忍，连续分歧说明系统可能不适合该候选人或议题设计有问题。

### 5.5 边界场景的动态处理

| 场景 | 检测方式 | 动态处理 |
|------|----------|----------|
| 候选人主动结束 | 回答含结束意图 | 立即路由至 REPORTING，标注未覆盖议题 |
| 答非所问 | Embedding 相似度低于阈值 | 首次提醒，累计 ≥ 3 次跳过议题 |
| 候选人断连 | 超时无响应 | 记录已完议题 → 生成部分报告 |

## 6. 工具调用

工具是 LLM Agent 与系统状态交互的**唯一入口**。

### 6.1 定义的工具

#### `parse_resume_document` — 简历解析工具

| 维度 | 说明 |
|------|------|
| 调用者 | Planner Agent |
| 时机 | 面试初始化阶段 |
| 输入 | PDF/Word 文件路径 |
| 输出 | 脱敏后的结构化候选人信息（skills, experience, education 等） |
| 调用位置 | `src/agents/planner.py`，Planner 初始化时 |

#### `submit_evaluation` — 评分提交工具

| 维度 | 说明 |
|------|------|
| 调用者 | Evaluator Agent（N=3 路各调用一次） |
| 时机 | 候选人完成作答后 |
| 输入 | score (1-100), topic_id, rationale (≥50字) |
| 输出 | 评分记录追加到 `evaluation_records` |
| 调用位置 | `src/tools/evaluation.py`，Evaluator 节点内部 |

### 6.2 工具调用约束

项目对工具调用做了**严格的参数校验**，不信任 LLM 的输出：

| 工具 | 校验项 | 失败处理 |
|------|--------|----------|
| `parse_resume_document` | 文件路径合法、格式正确（PDF/Word） | 抛 `ValueError`，使用默认候选人信息 |
| `submit_evaluation` | score ∈ [1,100]、topic_id 匹配当前议题、rationale ≥ 50 字 | 抛 `ValidationError`，该评分丢弃，不计入 σ |

这些校验通过 **Pydantic v2** 模型在工具入口处拦截非法数据。

### 6.3 工具调用链路

完整的工具调用流程：

```
简历上传 → parse_resume_document → candidate_info (脱敏)
    ↓
Planner 读取 candidate_info → 生成 interview_plan
    ↓
Questioner 读取 interview_plan + chat_history → 生成提问
    ↓
候选人作答 → 写入 chat_history
    ↓
Evaluator × 3 并行 → 各自生成评分
    ↓
submit_evaluation × 3 → Pydantic 校验 → evaluation_records
    ↓
Router 读取 evaluation_records → σ 分析 → 决策下一状态
```

### 6.4 工具 vs LLM 的关系

```
┌──────────────┐    校验层     ┌──────────────┐
│  LLM 生成内容  │ ──────────→ │  工具校验并写入  │
│  (非确定性)    │             │  (确定性)       │
└──────────────┘              └──────────────┘
```

LLM 只负责生成内容，**不直接修改状态**。所有状态写入必须通过预定义的工具，这确保了：

1. **可测试**：可以独立测试工具而不依赖 LLM
2. **可审计**：工具调用记录写入结构化日志
3. **安全**：非法数据在校验层被拦截，不会污染 σ 计算

## 7. 人类作为一等智能体

Supervisor 不是"调试工具"，而是协作链中的一等公民。当 AI 评估器们分歧过大时，系统**承认自身局限**，将决策权交给人：

```
ROUTING (σ > 15) → ESCALATED → 前端仲裁页面
                                       ↓
                              ┌─────────┼─────────┐
                              ▼         ▼         ▼
                           CONTINUE   SKIP       END
                              │         │         │
                              ▼         ▼         ▼
                          Questioner  下一议题  REPORTING
                          (当前议题)
```

人类输入 CONTINUE/SKIP/END 后，系统继续流转——人类和 AI 在同一个状态机中协作。

## 总结

本项目中多智能体协作、动态规划和工具调用的核心公式：

> **通过状态机编排 + 统计指标治理，让不同职责的 Agent 在受控的信息边界内独立工作，根据运行时数据动态调整策略，通过预设的工具契约安全地写入状态。**

具体实现：
1. **角色隔离** — 四个 Agent 各负其责，最小上下文传递
2. **状态共享** — InterviewState 作为唯一通信介质
3. **并行评估** — N=3 路独立打分，σ 量化分歧
4. **动态路由** — 根据 σ 值和计数器动态选择路径
5. **动态调整** — CoT 重试 → 降维追问，难度随表现变化
6. **工具契约** — LLM 生成 → 工具校验 → 状态写入
7. **人类协作** — 分歧过大时承认局限，交给人工裁决
